# ТЗ: Сайт-шежіре рода Аргын → Темеш → Манам

## 1. Общее описание

Веб-сайт, посвящённый истории и родословной (шежіре) рода **Аргын → Куандық → Темеш → Манам** (Средний жуз). Ключевая функция — интерактивное родовое древо, которое пополняется самими представителями рода: пользователи регистрируются, заполняют информацию о себе и своих предках и через **заявку** отправляют данные на модерацию. После одобрения администратором данные появляются в общем древе.

**Важная особенность казахского шежіре:** родословная ведётся строго по мужской линии (жеті ата). Это сильно упрощает модель данных — каждому человеку достаточно одного поля `father_id`, полноценная генеалогическая модель «два родителя + браки» не нужна. Древо — обычная иерархия (один корень — Манам или Темеш).

## 2. Технологический стек

| Слой | Технология |
|---|---|
| Frontend | React 18 + TypeScript + Vite |
| Стили | Tailwind CSS (mobile-first) |
| Роутинг | React Router v6 |
| Древо | `family-chart` (D3-based, MIT, TS-типы) — основной вариант; fallback: кастомный рендер на `d3-hierarchy` (т.к. древо однородительское, кастом прост) |
| Backend | Supabase: Postgres, Auth, Storage (фото), RLS |
| Деплой | Vercel (SPA), env-переменные `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY` |
| Языки интерфейса | Русский + казахский (i18next), переключатель в шапке |

## 3. Роли пользователей

1. **Гость** — просмотр древа, страниц персон, истории рода. Регистрация.
2. **Пользователь** — всё, что гость + личный кабинет, подача заявок на добавление/исправление персон, привязка себя к узлу древа.
3. **Модератор/Админ** — рассмотрение заявок (одобрить/отклонить с комментарием), CRUD персон напрямую, управление контентом страницы «История рода», назначение модераторов (только админ).

## 4. Модель данных (Postgres / Supabase)

```sql
-- профили (расширение auth.users)
create table profiles (
  id uuid primary key references auth.users on delete cascade,
  full_name text not null,
  phone text,
  role text not null default 'user' check (role in ('user','moderator','admin')),
  person_id bigint references persons(id), -- привязка "я в древе"
  created_at timestamptz default now()
);

-- персоны древа
create table persons (
  id bigint generated always as identity primary key,
  full_name text not null,
  father_id bigint references persons(id) on delete restrict,
  generation int, -- вычисляемое/денормализованное поле для сортировки
  birth_year int,
  death_year int,
  is_alive boolean default true,
  bio text,
  photo_url text,
  residence text, -- город/аул
  created_by uuid references profiles(id),
  is_verified boolean default false,
  created_at timestamptz default now(),
  updated_at timestamptz default now()
);

-- заявки
create table submissions (
  id bigint generated always as identity primary key,
  submitted_by uuid not null references profiles(id),
  type text not null check (type in ('add_person','edit_person','link_self')),
  target_person_id bigint references persons(id), -- для edit/link
  payload jsonb not null, -- поля персоны + father_id
  status text not null default 'pending' check (status in ('pending','approved','rejected')),
  moderator_comment text,
  reviewed_by uuid references profiles(id),
  reviewed_at timestamptz,
  created_at timestamptz default now()
);
```

**RLS-политики:**
- `persons`: SELECT — все (включая anon); INSERT/UPDATE/DELETE — только `role in ('moderator','admin')`.
- `submissions`: INSERT — авторизованный, `submitted_by = auth.uid()`; SELECT — автор видит свои, модераторы — все; UPDATE (статус) — только модераторы.
- `profiles`: пользователь читает/редактирует только свой, модераторы читают все. Поле `role` менять может только админ (отдельная политика или RPC).
- Роль хранить также в JWT custom claims либо проверять через `security definer`-функцию `is_moderator()`, чтобы не было рекурсии в политиках.

**Одобрение заявки** — Postgres-функция `approve_submission(submission_id)` (security definer): в транзакции создаёт/обновляет запись в `persons` из `payload` и меняет статус. Не делать это двумя запросами с клиента.

**Storage:** bucket `photos`, публичное чтение, запись — авторизованным, ограничение 2 МБ, ресайз на клиенте перед загрузкой.

## 5. Страницы

1. **Главная `/`** — герой-блок, краткая история рода (Аргын → Мейрам → Куандық → Темеш → Манам), CTA «Открыть древо» и «Добавить себя в древо».
2. **Древо `/tree`** — центральная страница:
   - зум/пан (мышь + touch: pinch-zoom обязателен);
   - клик по узлу → карточка (фото, годы, краткое био, ссылка на страницу персоны);
   - поиск по имени с центрированием на найденном узле;
   - фильтр «показать N поколений от узла» — критично для производительности при сотнях узлов;
   - кнопка «Я в этом древе» для авторизованных → заявка `link_self`;
   - неверифицированные узлы визуально отличаются (пунктир/бейдж).
3. **Персона `/person/:id`** — фото, годы жизни, биография, цепочка предков до корня («жеті ата»), список детей, кнопки «предложить исправление» и «добавить ребёнка» (обе → форма заявки).
4. **Auth `/login`, `/register`** — email+пароль и Google OAuth (Supabase Auth). После регистрации — онбординг: «найдите себя в древе или подайте заявку».
5. **Кабинет `/profile`** — мои данные, мои заявки со статусами, моя ветка древа.
6. **Подача заявки `/submit`** — пошаговая форма: выбор отца через поиск по древу → данные персоны → фото → предпросмотр → отправка.
7. **Админка `/admin`** — таблица заявок с diff-просмотром payload vs текущие данные, кнопки одобрить/отклонить + комментарий; управление персонами; управление контентом главной. Доступ по роли.

## 6. Древо: реализация

- Данные тянуть одним запросом `select id, full_name, father_id, photo_url, is_verified, birth_year, death_year from persons` (для шежіре в несколько сотен–тысяч узлов это нормально), строить иерархию на клиенте через `d3.stratify()`.
- Рекомендуемая библиотека — **family-chart** (D3, MIT, работает с React, есть TS-типы). Так как древо однородительское, допустим и лёгкий кастомный вариант: `d3-hierarchy` (`tree()` layout) + SVG + `d3-zoom` — меньше зависимость, полный контроль над карточками узлов.
- Виртуализация не обязательна на старте, но обязателен режим «свернуть/развернуть ветку» и ограничение глубины по умолчанию (например, 5 поколений от корня).

## 7. Адаптивность

- Mobile-first, брейкпоинты Tailwind (`sm/md/lg`).
- Древо на мобильном: полноэкранный режим, touch-жесты (pinch/pan), карточка персоны — bottom sheet вместо модалки.
- Формы — одна колонка на мобильном, крупные тач-таргеты (min 44px).
- Навигация: бургер-меню на мобильном.

## 8. Нефункциональные требования

- SEO: мета-теги, OpenGraph; страницы персон должны индексироваться (если критично — рассмотреть Next.js вместо Vite, но для MVP достаточно SPA + prerender главной).
- Языки: ru/kk через i18next, ключи с самого начала, без хардкода строк.
- Валидация форм: zod + react-hook-form.
- Защита от спама заявок: rate limit (не более N pending-заявок на пользователя).
- Индексы: `persons(father_id)`, `submissions(status)`, `persons using gin (to_tsvector('simple', full_name))` для поиска.

## 9. Этапы (для Claude Code)

1. **Setup:** Vite + React + TS + Tailwind + Router, Supabase-проект, схема БД + RLS + seed-скрипт с базовой цепочкой (Аргын → … → Манам и известные колена).
2. **Auth + профили:** регистрация, логин, кабинет.
3. **Древо (read-only):** загрузка персон, рендер, зум/пан, поиск, страница персоны.
4. **Заявки:** форма подачи, мои заявки, RPC `approve_submission`.
5. **Админка:** модерация, CRUD персон.
6. **Полировка:** i18n, адаптив, фото-загрузка, деплой на Vercel.

## 10. Исторический контекст для контента главной

По шежіре Аргын входит в Средний жуз; линия рода: Аргын → Мейрам → **Куандық** → Әмір Темір, от которого происходят роды Бөріші, **Темеш** и Қалқаман. Детьми Темеша в источниках называются Төбет, Қайнар, Қангелді, Қошқар, Жұман. **Место Манама в этой цепочке в открытых источниках не найдено** — эту часть шежіре нужно взять из семейных записей и заложить в seed-данные вручную. Из известных представителей подрода Темеш — Рахимжан Кошкарбаев, Народный герой Казахстана.
